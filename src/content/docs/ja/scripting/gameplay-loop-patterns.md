変数、`update()`、入力、`if` を学んだ後、多くの人が同じところで止まります。個々の知識は分かるのに、それらを本物のゲームプレイスクリプトへどう入れるかが見えないのです。

このページは、その中間を埋めます。高度な API に急いで進むことも、すぐ完全なプロジェクトを作ることも求めません。目標は 1 つだけです。「基礎構文」を明確なゲームプレイループに整理できるようにすることです。

## まず 1 文のゲームプレイルールから始める

スクリプトを書く前に、コードから考え始めないでください。まずゲームプレイルールを人間の言葉で書きます。

> プレイヤーが Fire を押したら発射する。発射後はクールダウンに入る。クールダウン中は再発射できない。HUD は現在状態を表示する。

この文は 4 種類の内容に分けられます。

| 種類 | 例 | スクリプトでは通常どこに置くか |
| --- | --- | --- |
| 入力 | `Fire` を押す | `update()` 内で `Input.pressed(...)` を読む |
| 状態 | クールダウン中か、残り時間はいくつか | スクリプトスコープの `var` |
| ルール | 発射可能な時だけ発射し、クールダウンが 0 になったら回復 | `if` 判断 |
| フィードバック | HUD テキスト、効果音、ログ | ルール変化後に更新 |

この表はコードより重要です。この 4 種類を分けられれば、スクリプトは最初から混乱しにくくなります。

## おすすめのスクリプト骨格

入門ゲームプレイスクリプトは、まずこの順序で書けます。

```swift
script SimpleAction {
    @export var cooldownTime: Float = 0.6

    var cooldown: Float = 0.0
    var status: String = "Ready"

    func start() {
        cooldown = 0.0
        status = "Ready"
    }

    func update(_ dt: Float) {
        // 1. 既存状態を更新する

        // 2. 读取输入

        // 3. ルールに基づいて状態を変える

        // 4. 给玩家反馈
    }
}
```

先にコメントを書いても構いません。コメントは飾りではありません。「毎フレーム何をするか」の順序を整える助けになります。

## 第 1 歩：まず状態が自然に流れるようにする

クールダウン、カウントダウン、継続時間のような状態は、プレイヤー入力がなくても変化します。通常、先にこれらを更新します。

```swift
if cooldown > 0.0 {
    cooldown -= dt

    if cooldown <= 0.0 {
        cooldown = 0.0
        status = "Ready"
    }
}
```

ここには 2 つの細部があります。

- `cooldown -= dt` は時間が減っていくことを意味します。
- 0 になったら `0.0` に丸めると、HUD と判断がきれいになります。

初心者は `cooldown -= dt` だけを書き、0 未満の処理を忘れがちです。短期的には動きますが、後の判断と表示が乱れやすくなります。

## 第 2 歩：次に入力を読む

入力は、それを使う場所の近くで読むのがおすすめです。最初に大量のキーを読み、数十行後に使う形は避けます。

```swift
if Input.pressed("Fire") && cooldown <= 0.0 {
    cooldown = cooldownTime
    status = "Fired"
    Audio.playTone("square", 520.0, 0.05, 0.2)
}
```

このコードはそのまま読めます。

1. このフレームで `Fire` が押された。
2. かつ現在クールダウン中ではない。
3. なら、クールダウンに入り、状態文字列を変え、効果音を鳴らす。

これは最もよくあるゲームプレイスクリプトの形です。入力は直接「作業」するのではなく、状態変化を引き起こします。

## 第 3 歩：HUD は最後に置く

HUD は通常 `update()` の末尾に置きます。そのフレームで計算し終えた結果を表示するためです。

```swift
ui.rect("action_panel", 12.0, 12.0, 280.0, 96.0, 0.03, 0.04, 0.06, 0.86)
ui.label("action_status", status, 24.0, 42.0)
ui.label("action_cooldown", "Cooldown: " + cooldown, 24.0, 70.0)
```

HUD を先に描いてから `status` を変えると、プレイヤーの見る内容が 1 拍遅れます。入門時はこの経験則を覚えてください。先にルールを計算し、最後に結果を表示します。

## 完全な小例

上の部分をまとめると:

```swift
script SimpleAction {
    @export var cooldownTime: Float = 0.6

    var cooldown: Float = 0.0
    var status: String = "Ready"

    func start() {
        cooldown = 0.0
        status = "Ready"
    }

    func update(_ dt: Float) {
        if cooldown > 0.0 {
            cooldown -= dt

            if cooldown <= 0.0 {
                cooldown = 0.0
                status = "Ready"
            }
        }

        if Input.pressed("Fire") && cooldown <= 0.0 {
            cooldown = cooldownTime
            status = "Fired"
            Audio.playTone("square", 520.0, 0.05, 0.2)
        }

        ui.rect("action_panel", 12.0, 12.0, 280.0, 96.0, 0.03, 0.04, 0.06, 0.86)
        ui.label("action_status", status, 24.0, 42.0)
        ui.label("action_cooldown", "Cooldown: " + cooldown, 24.0, 70.0)
    }
}
```

このスクリプトは小さいですが、完全なゲームプレイスクリプトの基本構造を含んでいます。パラメータ、状態、入力、ルール、フィードバックです。

## 1 つのアクションをゲームプレイへ拡張する

次にすることは、書き方を変えることではありません。同じ構造の中に状態を増やしていきます。

| 追加したいゲームプレイ | 新しい状態 | 新しいルール |
| --- | --- | --- |
| 弾薬 | `ammo` | 発射時に 1 減らし、弾薬がなければ発射できない |
| リロード | `reloading`、`reloadTimer` | Interact 後にカウントダウンし、終了後に補充 |
| 得点 | `score` | 命中または収集時に加点 |
| カウントダウン | `timeLeft`、`gameOver` | 時間が 0 になったら入力を止める |

たとえば弾薬を加える時、スクリプトを作り直す必要はありません。入力判断に条件を 1 つ足します。

```swift
if Input.pressed("Fire") && cooldown <= 0.0 && ammo > 0 {
    ammo -= 1
    cooldown = cooldownTime
    status = "Fired"
}
```

弾薬不足なら、別のルールを書きます。

```swift
if Input.pressed("Fire") && ammo <= 0 {
    status = "Empty"
    Audio.playTone("square", 120.0, 0.08, 0.18)
}
```

ゲームプレイスクリプトはこうして育ちます。巨大なシステムを一度に書くのではなく、明確な小さな状態と小さなルールを増やしていきます。

## `wait()` を使う時、タイマーを使う時

`wait()` は短く孤立したリズムに向いています。

```swift
canFire = false
wait(fireRate)
canFire = true
```

ただし、待っている間にも HUD を表示し続ける、プレイヤーが動ける、失敗を処理する、キャンセルできる、という場合は明示的なタイマーにします。

```swift
if reloadTimer > 0.0 {
    reloadTimer -= dt
    status = "Reloading"
}
```

経験則:

| 状況 | 推奨 |
| --- | --- |
| 単純なクールダウン例だけ | `wait()` でよい |
| HUD に残り時間を表示する | 明示的なタイマー |
| 待機中もプレイヤーが動ける | 明示的なタイマー |
| 状態がキャンセルまたは中断される可能性がある | 明示的なタイマー |

完全なプロジェクトチュートリアルでは、多くの場合明示的なタイマーを使います。デバッグしやすく、他システムとも合わせやすいためです。

## 長いスクリプトの読み方

数十行または数百行のゲームプレイスクリプトを見る時、最初の行から力任せに読まないでください。まず次を探します。

| まず探すもの | 分かること |
| --- | --- |
| `@export var` | このスクリプトで調整できる値 |
| スクリプトスコープの `var` | 覚えておく状態 |
| `start()` | 開始時の初期化 |
| `update()` 前半 | 毎フレーム自然に変化する状態 |
| 入力判断 | プレイヤーが引き起こせる状態変化 |
| HUD と効果音 | どのルールがフィードバックを出すか |

この読み方は次ページの高度なランタイム API に直接つながります。高度な API は別世界ではありません。「シーンを変える、シーンを検索する、音を鳴らす、HUD を描く」ための道具が増えるだけです。
